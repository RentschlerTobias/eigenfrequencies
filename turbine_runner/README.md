# turbine_runner

MVP (step 1): take a hydraulic turbine runner geometry generated by **dtOO** and
compute its structural **eigenfrequencies** with FEniCSx.

Step 2 ŌĆö avoiding a forbidden frequency band via optimization ŌĆö is **out of scope**
here.

## Two-container workflow

dtOO and FEniCSx live in different containers and never run together. They are
linked only by a `.msh` file in `data/` (the shared volume).

```
[dtOO container]                         [fenicsx container]
atismer/dtoo-opensuse:stable             docker/fenicsx.Dockerfile
   dtoo_export.py  ŌöĆŌöĆwritesŌöĆŌöĆŌ¢Č  data/runner.msh  ŌöĆŌöĆreadsŌöĆŌöĆŌ¢Č  main.py
```

### Stage 1 ŌĆö generate geometry (dtOO container)

Produces `data/runner.msh` from the dtOO `ruWithRounding_mechMesh` (the structural
mechanical mesh).

```bash
docker run --rm \
  -v "$PWD/turbine_runner/data:/work" \
  -v "$PWD:/src" \
  atismer/dtoo-opensuse:stable \
  python3 /src/turbine_runner/dtoo_export.py
```

XML config / state / case / volume names are overridable via env vars at the top
of `dtoo_export.py` (`DTOO_MACHINE_XML`, `DTOO_CASE_XML`, `DTOO_STATE`,
`DTOO_CASE`, `DTOO_MECH_VOLUME`).

> **Open item:** the tistos XML configs and the `axial_turbine_database` must be
> present in the dtOO container (historically in `block_structured_meshing`, git
> commit `eb33bff`, cloned at runtime). Confirm/locate before the first run.

> **Not the CFD mesh:** `block_structured_meshing/T1_9/T1_9_ru_gridGmsh.msh` is the
> *fluid* flow-channel mesh, **not** a structural solid. It is only usable as a
> readŌåÆassemble smoke-test, never for valid frequencies.

### Stage 2 ŌĆö eigenfrequencies (fenicsx container)

```bash
docker run --rm \
  -v "$PWD:/workspace" -w /workspace/turbine_runner \
  eigenfrequencies-fenicsx:latest \
  python3 main.py
```

(Build / launch the fenicsx image with `scripts/run_container.sh` /
`docker/fenicsx.Dockerfile`.)

## Set the boundary condition first

The dtOO mesh is in scaled, non-physical coordinates and its rotation axis is
**not** along x (unlike the beam demo). Run the axis-discovery diagnostic once
after the first export:

```bash
# inside the fenicsx container, in turbine_runner/
python3 mesh_prep.py
```

It prints the coordinate bbox and per-axis spans. Use them to set `BCConfig` in
`config.py`:

- `axis` ŌĆö the rotation axis (usually the longest span)
- `hub_radius`, `hub_center` ŌĆö radial clamp region around the axis
- `axial_min` / `axial_max` ŌĆö restrict the clamp to the hub bore band
- or `mode="axial_plane"` + `plane_value` to clamp a single plane

## Verification checklist

1. `data/runner.msh` exists and is non-trivial in size.
2. Element histogram shows volume cells (gmsh tet types 4 or 11). If only
   tri/quad appear, the `mesh_prep` volume-meshing fallback runs (needs a
   `step_path`, or a closed surface mesh).
3. `mesh_prep` logs `topology.dim == 3`.
4. Solver logs **fixed DOFs > 0** and a clamped-node bbox located at the hub.
5. Lowest ~6 frequencies are **real and positive** (near-zero values ŌćÆ rigid-body
   modes ŌćÆ the clamp failed; fix `BCConfig`).
6. `output/modes.xdmf` opens in ParaView with sensible blade deformation.
