## 1. Was das Projekt ist
Formoptimierung eines Francis-Turbinen-Runners per **Differential Evolution (DE)**.
Jede Design-Auswertung: **dtOO** (Geometrie + Mesh) → **FEniCSx/dolfinx** (modale
Eigenfrequenzen) → optional **OpenFOAM/simpleFoam** (CFD: Wirkungsgrad eta,
Kavitationsvolumen vcav, Förderhöhe dH). Ziel-Objective kombiniert CFD-Güte +
Resonanz-Penalty (Eigenfrequenzen aus verbotenem Band halten). Läuft verteilt auf
**bwUniCluster 3.0** via **Pyro5 RPC**: 1 Client (`optimize_de.py`) + N persistente
Worker (`server_de.py`), 1 Worker pro `srun`-Step.

Referenz-Projekt (bewährt, gleicher Cluster): `../de_framework` (Branch `bwCl`) —
pygmo-basierte DE, tistos-CFD-Maschinerie. Vieles hier ist davon abgeleitet.

## 2. Aktueller Zustand
**Funktioniert (verifiziert auf Cluster):**
- Pyro5 Worker-Discovery über Nodes (A2, file-based URIs) — 128/128 bzw. 256/256.
- dtOO + FEniCSx end-to-end, echte Eigenfrequenzen, resonance-only Objective (~5.3).
- Checkpoint/Resume, Live-Progress (tqdm + de_history.jsonl), Fitness-Metriken.

**Implementiert, NOCH NICHT auf Cluster verifiziert:**

  liest eta/vcav/dH. **Muss noch per Smoke-Run getestet werden** (siehe §6).

## 3. Architektur / Schlüsseldateien (alle unter `turbine_runner/`)
- `optimize_de.py` — DE-Client: URI-Discovery, DE-Schleife, ThreadPool-Dispatch an
  Worker, Checkpoint/Resume (`de_state.json`), Live-View (tqdm + `de_history.jsonl`),
  Fitness-Metriken pro Generation.
- `server_de.py` — Pyro5-Worker-Daemon. `Evaluator.evaluate(x, labels)`:
  `_run_dtoo` → `_run_fenicsx` → (CFD_ENABLED) `_run_cfd` + `evaluate_cfd` +
  `combined_objective`. Schreibt eigene URI atomar nach `$DE_URI_DIR/worker_<id>.uri`.
- `optimize.py` — Worker-Stages: `_run_dtoo` (dtOO-Container, Mesh),
  `_run_fenicsx` (enroot `pyxis_fenicsx`, modal), `_run_cfd` (NEU: OF-Case bauen +
  `sbatch.tistos_ru_of.sh`). `DTOO_FAIL_PENALTY = 1e6`.
- `dtoo_export.py` — dtOO Geometrie+Mech-Mesh (für FEniCSx), geometry-only.
- `dtoo_cfd_build.py` — NEU: schreibt `<state>.xml` aus templateState+Design,
  `CreateStates` + `CreateMeshes` → OF-Case `tistos_ru_of_n_<state>/`.
- `cfd_eval.py` — liest `postProcessing/<fo>/100` (Q_ru_in, ptot_ru_in/out, V_CAV,
  forces `moment.dat` total-z Index 3) → eta/vcav/dH. **Spaltenindizes noch gegen
  echten Cluster-Dump zu bestätigen.**
- `objective.py` — `combined_objective` (f_cfd + f_resonance), `resonance_term`.
- `config.py` — `DesignConfig` (3 DoF: `cV_ru_t_le_a_0.5`, `_mid_`, `_te_`),
  `DEConfig`, `CFDConfig` (post_folder="100", end_time=500, omega, rho, g).
- `cfd/tistos_files/` + `cfd/xml/` — NEU: aus de_framework kopierte tistos-OF-Case
  (machine.xml, templateState.xml, createStatesAndMeshes.py, sbatch.tistos_ru_of.sh,
  xml/tistos_ru_of.xml mit den functionObjects). `_run_cfd` staged sie pro Worker.
- `cluster/submit_de.sh` — SLURM-Batch: source ~/pe, Worker via `srun` verteilt,
  CFD-Env (FOAM_SIGFPE=0, OSLO_LOCK_PATH=/tmp), CFD_ENABLED default 1.

## 5. Offene Punkte / bekannte Themen
- **CFD-Verifikation (blockierend für CFD-Nutzung):** Namen der functionObjects
  bestätigt (Q_ru_in, ptot_ru_in/out, forces, V_CAV in `cfd/xml/tistos_ru_of.xml`),
  aber die exakten `.dat`-**Spaltenindizes** (v.a. `moment.dat` total-z) sind
  Annahmen. Erster CFD-Smoke-Run muss einen echten `postProcessing/`-Dump liefern →
  Spalten in `cfd_eval.py` abgleichen/fixen.
- **Generationen-Leerlauf (bekannt, kein Fix gewünscht):** synchrone DE-Barriere →
  Straggler-Tail (schnelle Worker idlen bis langsamster fertig) + Thundering Herd
  beim Gen-Start (viele Container/simpleFoam gleichzeitig). Lösung wäre **async
  steady-state DE** (rolling dispatch, wie de_frameworks pygmo-Archipel). User will
  das aktuell NICHT — nur dokumentiert.
- **Fehlende subprocess-Timeouts:** `_run_dtoo`/`_run_fenicsx` haben kein Timeout →
  ein hängender Container blockiert die ganze Generation. `_run_cfd` hat 1800s.
- **CFD-Ressourcen:** simpleFoam pro Worker braucht mehr `cpus-per-task` (MPI) und
  Produktions-Partition. `dev_cpu_il` (30 min) reicht nicht für volle CFD-Läufe.

## 6. Wie ausführen (Cluster, User st_ac136362)
Env: `source ~/pe` (py313-dtoo, dtOOPythonSWIG, pyDtOO, simpleFoam). FEniCSx via
enroot-Container `pyxis_fenicsx`. Shared-FS: `/pfs/work9/...`. Scratch: `$TMPDIR`.

**Resonance-only (funktioniert, schnell):**
```
sbatch --export=ALL,CFD_ENABLED=0 cluster/submit_de.sh
```
**CFD-Smoke (= Verifikation, ZUERST):**
```
git pull
sbatch --partition=cpu_il --nodes=1 --ntasks-per-node=2 --cpus-per-task=16 \
       --time=01:00:00 --export=ALL,DE_POP_SIZE=2,DE_MAX_GEN=1,DE_FRESH=1 \
       cluster/submit_de.sh
```
Danach prüfen: `server_logs/worker_0.log`; CFD-Logs `$TMPDIR/worker_0/cfd_build.log`
+ `cfd_solve.log` (rauskopieren); `turbine_runner/de_history.jsonl` → eta/vcav/dH
real (nicht 1e6)? Einen echten `postProcessing/`-Dump gegen `cfd_eval` abgleichen.
**Voller Lauf** danach: mehr Nodes + Generationen; Checkpoint sichert lange Läufe.

Wichtige Env-Vars: `CFD_ENABLED` (0/1), `DE_POP_SIZE`, `DE_MAX_GEN`, `DE_SEED`,
`DE_FRESH=1` (Neustart ohne Checkpoint), `DE_TQDM_INTERVAL`, `CFD_TIMEOUT`,
`W_RESONANCE`. Checkpoint: `turbine_runner/de_state.json`. Logs im `.out` +
`de_history.jsonl`.
